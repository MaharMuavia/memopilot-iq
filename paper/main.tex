\begin{abstract}
Large language model (LLM) assistants are stateless across sessions: they forget
user preferences, project decisions, and constraints, and continue to give
\emph{outdated} advice after a decision has changed. We present \sysname{}, a
persistent-memory agent organized around \osname{}, a self-contained memory
intelligence layer that sits between the conversation and the LLM. \osname{}
(i)~extracts \emph{typed, structured} memories from dialogue rather than storing
raw transcripts; (ii)~ranks candidates with a single, fully interpretable
linear scoring function that fuses semantic similarity with intrinsic signals
(importance, recency, confidence, usage), contextual signals (project match,
criticality), and explicit penalties (outdated, privacy, supersession);
(iii)~assembles the prompt under a strict token budget that \emph{guarantees}
inclusion of critical and pinned memories; (iv)~performs non-destructive
forgetting through expiry, archival, and supersession state transitions; and
(v)~emits a complete \emph{Memory Trace} that records, for every answer, which
memories were injected or skipped, their component scores, the reason, and the
token cost. We formalize the scoring and budgeting procedures and describe a
reproducible benchmark harness that contrasts the memory-augmented agent against
a no-memory baseline across six diagnostic scenarios spanning preference recall,
cross-session recall, contradiction/supersession, expired-deadline avoidance,
and critical-instruction retention. With a live \textsc{Qwen} backend
(\texttt{qwen3.7-max}) the memory agent reaches a task accuracy of $1.00$ versus
$0.33$ for the no-memory baseline while using a $2{,}500$-token context budget,
recalling target facts at \textsc{recall@5} $=1.00$, producing zero
outdated-memory leaks, and reducing injected context by roughly $97\%$ relative
to a full-history baseline. We argue that
\emph{interpretability and lifecycle management}, not raw capacity, are the
decisive properties for trustworthy long-term agent memory, and we release the
full system, benchmark, and trace format to support reproducible study.
\end{abstract}

\section{Introduction}
\label{sec:intro}

Transformer-based LLMs condition only on the tokens inside their context
window~\citep{vaswani2017attention,brown2020gpt3}. In a multi-session assistant
this produces a structural failure mode: once a conversation ends, everything
the user established is gone, including their stack preferences, the decisions
made for a project, hard constraints, deadlines, and past mistakes. Users are
forced to re-explain context at the start of every session, and, worse, the
assistant keeps repeating advice that a \emph{later} decision has already
overturned. A user who first chose React and later switched to Next.js will
keep receiving React advice, because the model has no notion that one statement
\emph{supersedes} the other.

Two families of remedies dominate practice. The first simply enlarges the
context window and replays long histories; but attention degrades over long
inputs~\citep{liu2024lostmiddle}, cost scales with length, and stale facts are
replayed alongside fresh ones with no mechanism to distinguish them. The second
is retrieval-augmented generation (RAG)~\citep{lewis2020rag}, which fetches
relevant passages from a vector store~\citep{karpukhin2020dpr,johnson2019faiss}.
RAG addresses \emph{capacity} but not \emph{lifecycle}: a vector store has no
opinion on whether a retrieved memory is still valid, whether it must always be
honored, or whether it should have been forgotten. It also typically offers no
account of \emph{why} a particular memory entered the prompt, an obstacle to
trust and debugging.

We take the position that the hard problems of agent memory are not capacity
problems but \emph{governance} problems: deciding what is worth remembering,
what must never be dropped, what has become invalid, and being able to
\emph{explain} every such decision. \sysname{} operationalizes this view through
\osname{}, a modular memory layer with five cooperating components (structured
extraction, interpretable scoring, hybrid retrieval, budget-aware assembly, and
non-destructive forgetting) unified by a transparent trace. The system runs in
two interchangeable modes: a fully offline local mode (SQLite with in-process
vectors and a deterministic embedding/LLM fallback) for reproducible evaluation,
and a cloud mode backed by managed object and table storage with Qwen
models~\citep{bai2023qwen,yang2024qwen2} for the language and embedding calls.

\paragraph{Contributions.}
\begin{enumerate}[leftmargin=1.4em,itemsep=2pt]
  \item \textbf{A typed memory schema and lifecycle.} A single record type with
  12 semantic categories, 6 lifecycle states, and three privacy levels that
  makes governance decisions first-class data rather than prompt heuristics
  (\S\ref{sec:schema}).
  \item \textbf{An interpretable scoring function.} A transparent linear model
  that fuses semantic, intrinsic, contextual, and penalty signals, with
  exponential recency decay and diminishing-returns usage scoring, so that every
  ranking decision decomposes into named, inspectable terms (\S\ref{sec:scoring}).
  \item \textbf{Budget-aware assembly with guaranteed criticality.} A context
  builder that fills a strict token budget by score while \emph{always}
  injecting critical/pinned memories, even when over budget, separating
  ``always honor'' instructions from ``recall if relevant'' ones
  (\S\ref{sec:budget}).
  \item \textbf{Non-destructive forgetting.} An engine that expires, archives,
  and supersedes memories through auditable state transitions, never silently
  hard-deleting user data (\S\ref{sec:forgetting}).
  \item \textbf{End-to-end transparency.} A Memory Trace that exposes the full
  decision surface of every answer, and a reproducible benchmark that quantifies
  recall, supersession, outdated-avoidance, accuracy delta, and token savings
  against a no-memory baseline (\S\ref{sec:trace}--\S\ref{sec:eval}).
\end{enumerate}

\section{Related Work}
\label{sec:related}

\paragraph{Memory for LLM agents.}
Interest in equipping LLM agents with long-term memory has grown rapidly; recent
surveys catalogue the design space~\citep{wang2024memoryos}. \emph{Generative
Agents}~\citep{park2023generative} introduced a memory stream scored by a
weighted sum of recency, importance, and relevance, with periodic reflection
that synthesizes higher-level observations. \osname{}'s scoring function is in
the same spirit but extends the signal set with confidence, usage, project
match, and, critically, a set of explicit \emph{penalties} for outdated,
private, and superseded content, turning lifecycle state into a ranking signal.
MemGPT~\citep{packer2023memgpt} frames the LLM as an operating system that pages
information between a fixed context window and external storage; \sysname{}
shares the budget-management goal but emphasizes an interpretable, typed scoring
policy and a guaranteed-inclusion rule for critical content rather than learned
paging. MemoryBank~\citep{zhong2024memorybank} applies an
Ebbinghaus-style~\citep{ebbinghaus1885memory} forgetting curve to memory
strength; we instead make forgetting a discrete, auditable \emph{state machine}
(expire/archive/supersede) so that every transition is explainable and
reversible. A-Mem~\citep{xu2025amem} builds dynamically linked memory networks;
our Memory Graph provides a comparable relational view (supersession and
related-to edges) but is derived from explicit lifecycle metadata.

\paragraph{Retrieval.}
Dense retrieval~\citep{karpukhin2020dpr} and approximate nearest-neighbor
indexes~\citep{johnson2019faiss} underpin most RAG systems~\citep{lewis2020rag}.
Sparse lexical scoring such as BM25~\citep{robertson2009bm25} remains a strong
complement, particularly for rare tokens and exact identifiers. \osname{} uses a
lightweight hybrid of dense cosine similarity and sparse keyword/tag overlap,
which is robust precisely on the short, entity-heavy memories typical of
developer assistants (framework names, deployment targets, file paths).

\paragraph{Reasoning and self-improvement.}
Our reflection pass, which merges duplicates, promotes frequently used
memories, and derives higher-level insights, is related to
Reflexion~\citep{shinn2023reflexion} and to chain-of-thought / ReAct style
prompting~\citep{wei2022cot,yao2023react}, but operates over the persistent
memory store rather than over a single episode's trajectory.

\paragraph{Positioning.} Relative to prior systems, \sysname{}'s distinguishing
commitment is \emph{transparency by construction}: scoring is a named linear
combination, inclusion decisions carry human-readable reasons, forgetting is a
visible state machine, and a complete per-answer trace is a first-class API
object. This makes the memory layer auditable and debuggable, which we argue is
a precondition for deployment in settings where wrong but confident recall is
costly. Table~\ref{tab:comparison} contrasts these governance properties across
representative memory systems.

\paragraph{Structured extraction.} Rather than storing raw turns, a
``Memory Editor'' prompt asks the LLM to return JSON describing
\texttt{new\_memories}, \texttt{updates}, and \texttt{forget} actions, each typed
and scored for importance and confidence. Secrets (API keys, tokens) are
redacted before storage. This converts free-form dialogue into a normalized,
queryable store and is what allows downstream scoring and forgetting to operate
on semantics rather than text spans.

\paragraph{Lifecycle as a state machine.} A memory is born \texttt{active} (or
\texttt{pinned} if the user pins it). It may be \texttt{superseded} by a later,
contradicting decision; \texttt{expired} when its deadline passes;
\texttt{archived} when stale and unused; or \texttt{deleted} only on explicit
user request. Crucially, retrieval excludes \{superseded, expired, deleted,
archived\} from normal context while preserving them on a timeline, so the
agent never acts on invalid information yet the history remains auditable.

\section{Interpretable Memory Scoring}
\label{sec:scoring}

Given a query $q$ with embedding $\mathbf{e}_q$ and a candidate memory $m$,
\osname{} computes a single scalar score as a weighted sum of named components.
Let $\mathrm{sim}(m,q)\in[0,1]$ be the hybrid similarity (\S\ref{sec:retrieval}),
$I_m,C_m\in[0,1]$ the importance and confidence, and define the derived signals

\begin{align}
\text{recency}(m) &= \exp\!\left(-\frac{\Delta t_m}{\tau}\right),
  \qquad \tau = 14~\text{days}, \label{eq:recency}\\
\text{usage}(m) &= 1 - \frac{1}{1 + \ln(1+u_m)}, \label{eq:usage}
\end{align}

where $\Delta t_m$ is the age in days since \texttt{updated\_at} and $u_m$ is the
usage count. Equation~\eqref{eq:recency} is an exponential forgetting curve with
a two-week half-life regime; Equation~\eqref{eq:usage} gives diminishing returns,
so a memory used many times is only modestly preferred over one used a few times.
Binary contextual indicators capture project match
($\text{proj}(m)=\mathbb{1}[\text{project}_m = \text{project}_q]$) and criticality
($\text{crit}(m)=\mathbb{1}[m~\text{is critical/pinned}]$). Penalty indicators flag
outdated content ($P^{\text{out}}_m$), privacy
($P^{\text{priv}}_m\in\{0,0.4,1\}$ for public/private/sensitive), and supersession
($P^{\text{sup}}_m=\mathbb{1}[\text{status}\in\{\text{superseded,expired,deleted}\}]$).
The final score is

\begin{equation}
\label{eq:score}
\begin{aligned}
S(m,q) =\ & 0.40\,\mathrm{sim}(m,q) + 0.20\,I_m + 0.15\,\text{recency}(m) \\
        & + 0.10\,C_m + 0.10\,\text{usage}(m) + 0.15\,\text{proj}(m) + 0.20\,\text{crit}(m) \\
        & - 0.30\,P^{\text{out}}_m - 0.25\,P^{\text{priv}}_m - 0.50\,P^{\text{sup}}_m.
\end{aligned}
\end{equation}

Three properties follow directly from the form of Equation~\eqref{eq:score} and
are central to the system's behavior:

\begin{description}[leftmargin=1.2em,itemsep=2pt]
  \item[Interpretability.] Every score decomposes into ten named terms. The
  trace (\S\ref{sec:trace}) reports each term, so a ranking can be explained and
  audited rather than merely trusted.
  \item[Semantics-led but not semantics-only.] Similarity is the largest single
  weight ($0.40$), so relevance dominates, but governance signals can override:
  a superseded memory pays a $-0.50$ penalty, larger than the full similarity
  weight, ensuring stale decisions sink below valid ones even when textually
  similar to the query.
  \item[Criticality as a soft-then-hard rule.] The $+0.20$ critical bonus raises
  critical memories in the ranking, and the budget stage (\S\ref{sec:budget})
  converts this into a \emph{hard} guarantee of inclusion.
\end{description}

The specific weights are hand-set priors chosen to encode these qualitative
commitments; we treat them as a transparent default policy and discuss
data-driven weight learning as future work (\S\ref{sec:limitations}).

\section{Hybrid Retrieval}
\label{sec:retrieval}

For each candidate, \osname{} blends a dense and a sparse signal. The dense
signal is cosine similarity between the query and memory embeddings, clamped to
$[0,1]$. The sparse signal is the fraction of query tokens that appear in the
memory's content or tags,
\[
\text{kw}(m,q) = \frac{|\,T(q)\cap T(m)\,|}{|\,T(q)\,|},
\]
where $T(\cdot)$ denotes the lowercased alphanumeric token set. The combined
similarity is
\begin{equation}
\mathrm{sim}(m,q) = \max\!\Big(\text{dense}(m,q),\ \tfrac{1}{2}\text{dense}(m,q) + \tfrac{1}{2}\text{kw}(m,q)\Big),
\end{equation}
which never underperforms pure dense retrieval and boosts memories with exact
lexical overlap, valuable for the identifier-heavy content of developer
assistants. Retrieval first applies structured filters (user, project, and the
exclusion of inactive states), scores all remaining candidates with
Equation~\eqref{eq:score}, and returns them sorted so that critical/pinned
records lead, then by score. Because memory stores per user/project are small
relative to web-scale corpora, exhaustive scoring is fast (sub-millisecond to
low-millisecond in local mode); an ANN index~\citep{johnson2019faiss} can be
substituted transparently behind the same interface at scale.

\section{Budget-Aware Context Assembly}
\label{sec:budget}

Retrieved candidates are not all injected. A context builder fills a strict
token budget $B$ (default $B=2{,}500$) under the following policy, with token
cost estimated at $\approx 4$ characters/token:

\begin{enumerate}[leftmargin=1.4em,itemsep=1pt]
  \item Order candidates with critical/pinned first, then by score $S(m,q)$.
  \item Inject critical/pinned memories \emph{unconditionally}, even if doing so
  exceeds $B$ (the budget is overridden, and the trace records this).
  \item Inject the top-$k$ ($k=8$) remaining memories in score order while they
  fit within $B$.
  \item Skip the rest, recording for each a reason, either ``token budget
  exhausted'' or ``outside the top-$k$ relevance window.''
\end{enumerate}

This cleanly separates two classes of memory with different service-level
requirements: ``always honor'' instructions (a security constraint must never be
dropped because the budget is tight) versus ``recall if relevant'' facts
(competing for the remaining budget by score). The assembled system prompt wraps
the selected memories in an explicit \texttt{ACTIVE LONG-TERM MEMORY} block with
instructions to honor critical items, prefer the most recent decision on
conflict, and never use superseded/expired content, reinforcing at the prompt
level the policy already enforced at the data level.

\section{Non-Destructive Forgetting}
\label{sec:forgetting}

A forgetting sweep runs over the store and applies three rules, each emitting an
auditable timeline event:

\begin{itemize}[leftmargin=1.4em,itemsep=1pt]
  \item \textbf{Expire.} A memory with a past \texttt{expires\_at} transitions
  to \texttt{expired} (e.g. ``the staging server is down today only'').
  \item \textbf{Archive.} An \texttt{active}, non-critical memory unused for more
  than 30 days with importance below $0.35$ transitions to \texttt{archived}.
  \item \textbf{Supersede.} When extraction detects a contradicting decision, the
  old record is linked via \texttt{superseded\_by} and excluded from retrieval.
\end{itemize}

No rule hard-deletes data; deletion happens only on explicit user request.
Expired/archived/superseded records remain queryable on the timeline and in the
Memory Graph but are excluded from context, so the agent never reasons from
invalid memory while the user retains a complete, reversible history. This is the
operational realization of our thesis that forgetting should be \emph{governed
and explainable} rather than lossy.

\section{Transparency: The Memory Trace}
\label{sec:trace}

Every chat response carries a \texttt{MemoryTrace} object recording the complete
decision surface: the list of \emph{included} memories with their full component
breakdown and inclusion reason, the list of \emph{skipped} memories with their
reason, the token budget and tokens used, the number of candidates considered,
and retrieval latency. The trace is rendered in the UI but is also a plain API
object, making the system debuggable and auditable end-to-end. To our knowledge
this degree of built-in inclusion-level explanation is uncommon among
memory-augmented assistants and is, we argue, the practical embodiment of the
interpretability the scoring function was designed to provide.

\section{Evaluation}
\label{sec:eval}

\subsection{Protocol}
We implement a benchmark harness that runs six diagnostic scenarios
(Table~\ref{tab:scenarios}). Each scenario seeds one or more \emph{setup
messages} (which \osname{} converts to memories), then poses a \emph{test
question}. The harness evaluates two systems on the identical question:

\begin{itemize}[leftmargin=1.4em,itemsep=1pt]
  \item \textbf{Memory agent.} The question is answered with the \osname{}-built
  context (retrieve $\rightarrow$ score $\rightarrow$ budget assembly).
  \item \textbf{No-memory baseline.} The same model answers with a bare system
  prompt and no long-term memory, isolating the contribution of the memory
  layer.
\end{itemize}

Correctness is decided by keyword criteria attached to each scenario:
\emph{expected} keywords that must be recalled and \emph{forbidden} keywords that
must not leak. To avoid penalizing legitimate references to a superseded choice
(e.g. ``use Next.js \emph{instead of} React''), a forbidden phrase counts as a
\emph{leak} only when it appears in an injected memory that does not also carry an
expected phrase, i.e. a genuinely outdated memory rather than a superseding
decision. From these we compute task accuracy, \textsc{recall@5}, supersession
success, outdated-memory leaks, an accuracy delta over baseline, and token
savings versus a full-history baseline. The headline numbers are from a live
\textsc{Qwen} backend (\texttt{qwen3.7-max}); the harness also runs against a
deterministic offline backend that reflects the memory \emph{mechanism} in
isolation and is exactly reproducible without credentials, so reported behavior
does not depend on a specific model endpoint.

\subsection{Analysis}
The accuracy gap is concentrated in exactly the scenarios the baseline cannot
solve in principle: cross-session recall and supersession require state that a
stateless model does not have, so the baseline's successes come only from
generic priors (e.g. naming a popular framework) rather than from the user's
actual choices. The zero-leak result is a direct consequence of the lifecycle
design: superseded and expired records are excluded at retrieval, so they cannot
reach the budget stage. The large token reduction (about $97\%$ on these
scenarios) shows that selective, scored injection under a budget retains the
relevant facts while discarding the bulk of raw history, addressing both cost
and the ``lost-in-the-middle'' degradation of long
contexts~\citep{liu2024lostmiddle}.

\section{Limitations and Future Work}
\label{sec:limitations}

\paragraph{Benchmark scope.} The six-scenario harness is diagnostic rather than
large-scale; its keyword-based correctness, while precise and reproducible, does
not capture answer fluency or subtle reasoning, and the reported numbers are
obtained under a deterministic offline backend chosen for reproducibility. A
natural next step is a larger, multi-session benchmark with live models and
human or model-graded answers, and comparison against external memory systems
(MemGPT~\citep{packer2023memgpt}, MemoryBank~\citep{zhong2024memorybank},
A-Mem~\citep{xu2025amem}) on a shared protocol.

\paragraph{Hand-set weights.} The scoring weights in Equation~\eqref{eq:score}
are interpretable priors, not learned. Learning them, e.g. by optimizing
downstream task accuracy or from preference feedback, while preserving the
linear, inspectable form is an attractive direction, as is per-user adaptation.

\paragraph{Extraction reliability.} Memory quality is bounded by extraction
quality; erroneous or over-eager extraction propagates downstream. Confidence
calibration, extraction-level evaluation, and user-in-the-loop confirmation for
low-confidence or sensitive memories would strengthen robustness.

\paragraph{Token estimation and budgeting.} The $\approx\!4$-characters/token
heuristic is adequate for budgeting but approximate; exact tokenizer-aware
accounting and adaptive budgets per model would tighten the guarantees.

\section{Conclusion}
\label{sec:conclusion}

We presented \sysname{}, a persistent-memory agent built on \osname{}, a memory
layer that treats long-term memory as a \emph{governance} problem rather than a
capacity problem. By making memory typed, scoring it with a fully interpretable
linear model, assembling context under a budget that guarantees critical
inclusion, forgetting through an auditable state machine, and exposing a complete
per-answer trace, the system remembers what matters, refuses to act on what has
become invalid, and can explain every decision. On a reproducible diagnostic
benchmark with a live \textsc{Qwen} backend it triples task accuracy over a
no-memory baseline ($1.00$ vs.\ $0.33$) with zero outdated-memory leaks while
cutting injected context by roughly $97\%$. We
release the system, benchmark, and trace format to support reproducible research
on trustworthy agent memory.
